# Prax Development Workflow

1. 先读上下文，再下结论。
2. 先确认复用路径，再新增实现。
3. 先补最小可执行检查链，再扩展策略门。
4. 原生运行时能力和 Claude Code 集成能力是两类语义，禁止混用语义。

---

## 强制检查点

### 检查点 1: 创建新文件/脚本前

- 用 Grep/Glob 搜索项目中是否已有类似工具
- 检查 package.json / pyproject.toml / Cargo.toml 是否已有相关依赖
- 用户提到"项目已有 X"时，必须先去读取 X 的实现

### 检查点 2: 连续 3 次修改同一处代码

触发条件（满足任一）：
- 同一文件的同一属性被修改 3 次
- 同一问题被用户提及 3 次
- 修改值在两个值之间反复横跳

触发后动作：
1. **停止修改**
2. 输出修改历史（每轮的旧值、新值、理由、验证结果）
3. 分析根因（缺失的验证、错误的假设）
4. 向用户报告，请求新的方向或验证方法

### 检查点 3: 声明"已完成"/"已修复"前

- 提供可验证的输出（截图、测试结果、diff、量化指标）
- 不能只说"刷新浏览器看效果"、"应该通过"
- 参见 `testing.md` 中"真实结果"的定义
