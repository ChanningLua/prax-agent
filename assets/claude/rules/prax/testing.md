# Prax Testing

- 变更后至少运行与改动直接相关的自动化检查。
- 新功能和缺陷修复优先采用先测后改。
- 若仓库已有 build / typecheck / test 入口，优先用现成入口，不重复发明检查链。
- 不能只声明”应该通过”，必须提供真实结果或明确说明未运行原因。

---

## “真实结果”的定义

“真实结果”必须是以下之一：

1. **命令输出**：完整的 stdout/stderr，包含退出码
2. **截图对比**：设计稿 vs 实现，标注差异和量化偏差
3. **量化指标**：具体的数值（如”还原度 92%，23/25 项通过”）
4. **Diff 输出**：代码变更的具体内容

### 禁止的”伪结果”

- ❌ “刷新浏览器看效果”
- ❌ “已同步到 X”
- ❌ “应该通过了”
- ❌ “看起来对了”

### 正确的结果报告

- ✅ “测试通过：`npm test` 输出 5 passed, 0 failed”
- ✅ “截图对比：间距偏差从 15px 降至 2px”
- ✅ “还原度 95%（19/20 项通过），失败项：皇冠位置偏差 3px”
