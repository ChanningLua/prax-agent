# Prax Coding Style

- 保持修改最小化，不为了”顺手优化”扩大改动面。
- 优先抽稳定接口，再改实现细节。
- 避免无边界的 `any`、占位实现和无证据推断。
- 非 trivial 任务必须留下可验证的检查路径。

---

## “证据”的定义

“避免无证据推断”中的”证据”必须是以下之一：

1. **可执行的验证**：运行命令得到的输出
2. **可视化的对比**：截图、diff、图表
3. **可追溯的数据源**：DSL 数据、API 响应、日志文件

### 示例

❌ **错误：凭感觉调整 CSS**
```css
/* 皇冠图太大了，改小一点 */
.member-deco { width: 60px; }  /* 猜测值 */
```

✅ **正确：基于 DSL 数据**
```css
/* DSL: 0:8069 容器 95x95, image 77x72 at (4.9, 5.3) */
/* 计算：95/2 = 47.5px, 但 image 实际宽度 77/2 = 38.5px */
.member-deco { width: 95px; }  /* DSL 容器尺寸 */
```

❌ **错误：连续猜测**
```
Round 1: width: 95px  // 太大
Round 2: width: 77px  // 还是大
Round 3: width: 60px  // 再小点
Round 4: width: 50px  // 继续猜
```

✅ **正确：截图验证 + 量化偏差**
```
Round 1: width: 95px → 截图对比：偏差 +20px
Round 2: width: 75px → 截图对比：偏差 +5px (在容差内) ✅
```
